Fase 0 · Saneamiento

2 sprints. Pre-requisito de todo lo demás. Freeze de cambios funcionales en backend financiero/referral.
5 tareas · ~2 sprints · 1 dev backend senior

Objetivo

Convergir las dos implementaciones paralelas de referidos, los dos webhooks Stripe, y los dos sistemas de balances (UserEntity vs WalletEntity) a una única fuente de verdad.

Entregables

  • Módulo src/backend/wallet/ independiente (extraído de finance/).
  • Directorio src/backend/referral/ (singular) eliminado, todo en referrals/.
  • Webhook único registrado en Stripe Dashboard.
  • Migración Firestore: balances de UserEntity → WalletEntity.
  • Smoke test E2E manual: checkout → webhook → wallet actualizado.

Criterios de aceptación

  • grep -r "src/backend/referral/" src/ no devuelve resultados.
  • Solo un endpoint webhook está registrado en Stripe (consola).
  • UserEntity ya no expone balanceComisionesPendientes ni balanceIngresosPendientes.
  • Decisión documentada en CLAUDE.md.

Fase 4.1 · Email transaccional

MVP de comunicación. Desbloquea el valor del booking ya construido.

JTBD

"Cuando un paciente paga una sesión, quiero que reciba la confirmación por email sin que yo tenga que hacer nada, para no parecer un negocio improvisado."

Scope

  • Sistema de eventos de dominio (booking.confirmed, booking.cancelled, enrollment.created).
  • Wrapper Nodemailer con SMTP del creador (si configurado) o fallback plataforma.
  • 4 plantillas Natureza: confirmación booking (paciente/creador), inscripción curso, cancelación.
  • Settings creador en /dashboard/settings: SMTP, sender, "Enviar correo de prueba".
  • EmailLogEntity para trazabilidad.

Edge cases

  • Creador sin SMTP → fallback plataforma con noreply@consciousness.class.
  • Email inválido → notificación in-app al creador, no bloquea booking.
  • Reintento exponencial 3x · 4º fallo emite NotificationEntity.

Métricas norte

  • % bookings con email entregado < 5min: >95%
  • Bounce rate: <2%
  • Tickets "no recibí confirmación": −80% vs baseline

Fase 5.1 · Cierre del bucle de Booking

~10 días. Cierra el feature ya 85% construido.

Scope

  • MUST: vista paciente /dashboard/learning/my-bookings.
  • MUST: vista creador /dashboard/coaching/bookings (calendar + lista).
  • MUST: cancelación con regla 24h (refund full vs no refund).
  • SHOULD: estado no-show + UI marca tras +30min del slot.
  • SHOULD: UI excepciones (vacaciones) en availability.
  • COULD (post-MVP): Google Calendar OAuth + Meet automático.

Reglas duras (no "debería", "debe")

  • Transición scheduled → cancelled con <24h debe marcar refundEligible: false.
  • Slot no debe ser reservable si solapa con otro booking scheduled (incluido durationMinutes).
  • no-show solo es transición válida desde scheduled y solo después de slot.endTime.

Fase 3 · Referidos multi-nivel + Stripe Connect Express

~4 sprints. Pre-requisito Fase 0.

3.A — Política asignable

  • UI dropdown "Política de Referidos" en cada builder de activo.
  • Validación dura tier1 + tier2 + plataforma ≤ 100%.
  • Política con uso > 0 no eliminable.
  • Snapshot de política en Booking/Enrollment al crearse (immune a cambios futuros).

3.B — Adaptador Stripe Connect Express

  • Implementar PaymentGatewayInterface con primer adaptador real.
  • Modelo: destination charges con application_fee_amount + transfers a afiliados.
  • Onboarding creador: /dashboard/settings/payouts → redirect Stripe Express.
  • Medidas de seguridad: 3DS obligatorio, rate-limit en endpoint payout (1 req/min/uid), webhook signature verification, idempotency keys, monitoreo de payouts > €500 con alerta superadmin.

3.C — Payouts reales

  • Cron semanal cuando saldo ≥ €X (configurable).
  • UI /dashboard/finances: disponible · pendiente · histórico.
  • Estado fiscal (residencia + datos KYC delegados a Stripe) bloqueante para payout > €1k.

Edge cases críticos

  • Reembolso post-comisión a afiliado → descuento en próximo payout (no clawback retroactivo).
  • Afiliado sin Connect → comisiones se acumulan en pendingBalance.
  • Cadena circular A→B→A → validación al asignar referredBy.

Fase 4.2 · CRM + WhatsApp Business

~4 sprints. Pre-requisito Fase 4.1.

Scope

  • Entidad Contact agregada (inscripciones + bookings + leads manuales) — vista unificada por persona.
  • Segmentos componibles: "inscritos curso X", "compraron en últimos 30d", "no abrieron último email".
  • Editor de campañas (no transaccional): asunto + plantilla + segmento + cola con rate-limit.
  • WhatsApp Business API directo (Meta) por número del creador. Onboarding incluye verificación + plantillas aprobadas (HSM).
  • Trigger automatizado: "estudiante sin progreso 7 días" → email empático opcional.

Decisiones de producto

  • WhatsApp: usamos solo plantillas pre-aprobadas para mensajes proactivos (24h window de Meta).
  • Privacidad: creador no puede exportar lista de contactos sin double opt-in del usuario.

Fase 6.2 · RAG Companion

~4 sprints. Stack: Firebase Vector Search (Firestore + extensión).

Política dura

  • Si no encuentra contexto suficiente → "Esto no está cubierto en el material del curso. Pregunta a tu creador."
  • Cita siempre fuente: lección + timestamp/página.
  • Disclaimer permanente: no consejo médico/psicológico.
  • Hard cap por creador. Al alcanzar: widget muestra "Vuelve mañana".

Pipeline indexación (MVP solo PDF)

  1. Subir PDF → cola async.
  2. Chunk por sección/página (overlap 100 tokens).
  3. Embedding con text-embedding-004.
  4. Persistir en Firestore Vector Search con metadata (lessonId, page).
  5. Query: top-K=5 por similitud + reranking.

Fase 6.3 · Agente Secretarial AI

~3 sprints. Pre-requisito: 4.2 + 5.1 + 6.2.

Tools del agente

  • getAvailableSlots(creatorId, dateRange)
  • createBooking(slotId, patientName, patientEmail) → genera link de pago
  • cancelBooking(bookingId) con verificación de identidad

Reglas duras

Agentes en paralelo

Cómo coordinar varios agentes Claude Code trabajando en ramas distintas sin colisionar.

Convención de ramas

PatrónUso
feature/T<phase>-<slug>Tareas funcionales nuevas. Ej: feature/T0.1-extract-wallet.
refactor/T<phase>-<slug>Saneamiento sin cambio funcional. Ej: refactor/T0.2-consolidate-referral.
chore/<slug>Config, scripts, infra (no produce features de cara al usuario).

Reglas para agentes en paralelo

  • Una rama = un agente = una tarea atómica del listado.
  • Antes de despachar: confirmar que las tareas depende de ya están merged a develop.
  • Cada agente abre PR contra develop, no contra master.
  • Tareas que tocan el mismo módulo backend (ej. dos tareas de Fase 3 en backend/payments/) no se paralelizan — se serializan.
  • Cada agente debe correr npm run typecheck antes de cerrar (no build — recordatorio en CLAUDE.md).

Aislamiento de worktree

Para evitar que dos agentes pisen el mismo working directory, usar git worktrees:

git worktree add ../cc-T0.1 -b feature/T0.1-extract-wallet
git worktree add ../cc-T6.1.1 -b feature/T6.1.1-journal-entity

Cada worktree es un directorio distinto con su propia rama. Lanzar el agente Claude Code dentro de cada worktree.

Falta cubrir (más allá del PM)

Áreas que no están en el roadmap actual y deberían añadirse antes de producción.

UI/UX

  • Design system formal: tokens (espaciado, motion, radii) sin documentar. Componentes shadcn instalados pero sin guía de uso consistente.
  • Onboarding del creador: el momento "signup → primer asset publicado" no tiene flujo guiado. Punto de mayor abandono.
  • Onboarding Stripe Connect: drop-off conocido en la industria. Necesita copy + UI cuidados (Fase 3).
  • UX conversacional: RAG widget y agente secretarial necesitan diseño dedicado (turnos, errores, citas, fallbacks).
  • Empty states + AI loading: latencia 2-5s requiere skeleton/streaming feedback.
  • Mobile-first audit: los terapeutas usan móvil; verificar dashboard + builders.
  • Accesibilidad WCAG AA: audiencia mental health = más crítica. Auditoría con axe-core.

Legal & Compliance

  • GDPR + datos de salud especiales: el journal entra en Art. 9 GDPR. Necesita base legal explícita y DPIA.
  • Términos & Privacidad: actualizar para reflejar uso de IA, retención, sub-procesadores (Google, Stripe, Meta).
  • Política de retención: ¿cuánto tiempo guardamos journal/chats? ¿cómo se borra?
  • Disclaimer médico/psicológico visible y aceptado al onboarding del estudiante.
  • DPA con Google (Vertex/Gemini), Stripe, Meta.

Trust & Safety

  • Crisis detection en journal y chat (mencionado en Fase 6).
  • Moderación community: Fase Core ya tiene Community pero no hay reportes/bloqueo de usuarios.
  • Verificación creador: ¿cómo evitamos que cualquiera se haga pasar por terapeuta? KYC ligero antes de publicar coaching.
  • Age gating: producto no apto para menores; verificar.

Datos & Operaciones

  • Analytics + event tracking: sin esto las métricas norte que definimos no se pueden medir. Recomendación: PostHog (open source, on-premise).
  • Testing strategy: el repo no tiene runner. Para pagos + booking + AI es crítico antes de producción. Mínimo: Vitest + Playwright para flujos críticos.
  • Observabilidad: logs estructurados, métricas, alertas. Stack: Cloud Logging + Cloud Monitoring (consistencia GCP).
  • Backups & disaster recovery: Firestore export programado; runbook ante pérdida de datos.
  • Performance SLOs: p99 booking confirmation, latencia AI streaming, tiempo de respuesta RAG.
  • i18n: español primero, pero strings hardcodeadas. Si se planea catalán/inglés/portugués brasileño, extraer ahora.

Negocio

  • Pricing del producto plataforma: ¿cobramos % de transacción? ¿tier mensual al creador? ¿free tier? Afecta arquitectura billing.
  • Soporte: ¿quién responde tickets de creadores? ¿Status page pública?
  • Help center / docs públicas para creadores y estudiantes.
  • Programa de referidos plataforma (distinto del de creadores) para crecimiento.
